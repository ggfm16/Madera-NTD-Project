
# Madera NTD Project
Analyzes 10 years of City of Madera transit data using a Python ETL pipeline across 70+ NTD Excel files. Includes QGIS geospatial visualizations using GTFS and 2020 census data, an interactive Excel dashboard, and findings on transit efficiency, ridership trends, and HSR readiness ahead of a shifting policy landscape.

## ETL Pipeline
The first step in this project was to create an ETL pipeline that could extract, transform, and load transit data from the National Transit Database (NTD) for the City of Madera (NTD ID:90199) from fiscal years 2014-2024.
### The Challenge
NTD data is gathered into collections of annual Excel files, each yearly collection is made up of different report types, such as operating expenses, service data, fare revenue, fleet inventory, and agency information. The data itself is publicly available, however it is not packaged for analysis. File naming conventions, column names, and sheet structure can vary from year to year. Processing this data manually would risk higher rates of error and would be a significant time-sink.
### How the Pipeline Works
The pipeline is organized around a central function that processes one year at a time and returns a list of rows, one row per mode of transit. For Madera this means that there are two rows, one for Fixed Route (FR) and another for Demand Response (DR). For each year, the pipeline uses a pattern-matching system that accounts for naming inconsistencies across years. Then it loads each file with a normalization layer that handles buried headers, inconsistent column names, and other idiosyncrasies. 
Once loaded, the pipeline filters each file for Madera's NTD ID and extracts the relevant fields: operating expenses, fare revenue, vehicle revenue miles, vehicle revenue hours, unlinked passenger trips, vehicles operated in maximum service, and fleet inventory. From these raw fields it derives a set of key performance indicators including cost per trip, cost per revenue mile, cost per revenue hour, ridership per revenue hour, farebox recovery percentage, and average fleet age. For years where Revenue Sources data is available, it also extracts the funding breakdown across fare, federal, state, and local sources.
2014 was processed without funding source data due to differences in file structure impeding processing, funding data was not considered a core metric for the scope of this project and was not included for 2014.
### Output
The pipeline produces a single CSV file with 22 rows and 22 columns covering both Fixed Route and Demand Response service for each year. The output has zero null values across all core metrics with the only null values being the funding data from fiscal year 2014. The CSV was audited against NTD produced reports and the source files themselves to ensure accuracy. This CSV file serves as the primary source for the Excel dashboard and analytical findings in this project.
### Running the Pipeline
The pipeline is designed to run in Google Colab with files organized in Google Drive. To run it yourself you can just upload the "NTD project" to the "My Drive" directory and it should work seamlessly. If you'd like to upload it to another directory you would need to download the "NTD project" from the repository and update the DATA_Folder path in the configuration block with the appropriate path information. 

## Excel Dashboard
The Excel dashboard provides an interactive single sheet interface for exploring Madera transit's KPIs across both Fixed Route and Demand Response service from 2014 to 2024. The dashboard is driven by an unpivoted Power Query Table and linked Pivot Tables, utilizing slicers to switch metrics with ease.
### Data Architecture
The CSV output from the ETL pipeline was loaded into Excel and then converted to a table. It was then connected via Power Query, meaning that any new columns added to the table will automatically propagate to the dashboard upon refresh.
The raw dataset is in CSV format, to make this compatible with slicers, the data was transformed in Power Query using an unpivot operation. The identifier columns, Year, Agency and Mode, were kept as is. Metric columns were consolidated into two new columns, "KPI_Type" which contained the metric name and "Value" containing the corresponding value figure. Irrelevant or highly granular metrics such as funding breakdown were filtered out within Power Query to avoid clutter. This Unpivoted table was moved to a hidden "Pivot Engine Room" along with the Pivot Tables that power the visualizations. 
### Dynamic Chart
The main visualization is a dual-line chart comparing Fixed Route and Demand Response trends over time for whichever metric is selected. The chart title updates dynamically to match the selected metric using a formula cell that mirrors the Slicer selection. Because the chart axis must accommodate both large volume metrics such as operating expenses in the millions and rate metrics such as farebox recovery expressed as a percentage, a custom Y-axis format code was applied that forces large numbers to display with comma separators, small decimals to render as percentages and suppresses sole zeroes to keep the axis consistent. 
### Interactive Table
A mirrored table built from a Pivot Table displays the exact figures for Fixed Route and Demand Response for each year, their totals and the absolute difference between the two modes. The table uses several layers of conditional formatting to respond to slicer inputs in real time.
When Farebox Recovery Percentage is selected, the table automatically overrides its default number formatting to display values as percentages. For all other metrics it defaults to comma-separated whole numbers. This formatting switch is driven by a formula that reads the associated Slicer's state through the Pivot Tables filter acting as an anchor cell. 
The difference column uses conditional formatting formulas to compare the two modes row by row, it shades the higher value's cell in the corresponding mode color (orange for Fixed Route and blue for Demand Response). This ensures that for each metric, the dominant mode can be identified. In addition to this, I used Excel's Top and Bottom rules to identify the all time Minimum and Maximum values for the selected metric. The formatting applies diagonal line patterns and bold font to the difference column and light fill colors to the mode total columns. Red for the minimum and green for the maximum. This allows for the quick identification of peak and down years, which years a particular mode outperformed the other and by how wide a margin, and identifying which years that margin was highest and lowest. All at a glance. These rules are layered in the conditional formatting hierarchy so they display simultaneously without conflicting, allowing a single cell to communicate multiple data points at once.

## QGIS Visualizations
The QGIS portion of this project provides spatial context for Madera's transit system by mapping route coverage against population distribution across the city. Two data sources were used: GTFS data from the Mobility Database which provided route and stop geometry, and 2020 Census Block shapefiles for Madera County which provided population and housing unit counts at the block level. Census block geometries were processed through QGIS geometry repair tools prior to analysis to resolve self-intersecting geometries in the source shapefiles. The project elements were also converted to a projected CRS as needed to maintain measurement accuracy.
### Fixed Route and Stop Map
The first visualization layer plots the City of Madera's four fixed route lines and all associated stops against a basemap of the city. Each route is color coded to match the agency's published system map. This provides a baseline reference for the system's footprint.
### Population Heatmap
The second visualisation layer overlays a kernel density heatmap of population derived from census block centroids weighted by total population (POP100). When toggled to be visible it gives an at a glance view of where population is concentrated relative to the system's layout without the noise introduced by block level boundaries. It is intentionally meant to provide an approximation of where population clusters are located meant for macro-level viewing.
### Population Density Choropleth 
The third and final visualization layer is more precise, taking advantage of the granularity provided by census blocks.
This layer is a graduated choropleth of population density per square mile, which was calculated from block geometry using the expression POP100 / ($area * 0.000000386102) to convert meters to miles. Blocks are styled according to widely accepted density thresholds for public transit: Blocks below 3,000 people per square mile are colored white (but left unselected in the checklist so that they appear transparent), blocks between 3,000 and 10,000 are rendered in a light red, and blocks above 10,000 are colored a dark red (a common benchmark for high-frequency transit viability). 
Labeling is scale dependent and rule based. Labels are suppressed at the city wide scale to preserve readability and only render at the neighborhood scale, when sufficiently zoomed in each block displays both its total population and calculated density. Labels are constrained to only render within their block boundary to avoid confusing overlap. 
I decided to include both population metrics because high density does not necessarily correlate to sustainable ridership populations, so showing both numbers can avoid flawed decision making based only on density. A link to a cloud hosted version of the map is included in the repository with all features fully intact.

## Presentation
The PowerPoint deck serves as the TLDR for this project, packaging the key visualizations and findings into a self-contained document that can be  read without cloning the repository or opening any other files. It is structured in two sections: a QGIS that provides spatial context for the system, and an NTD data section that goes through a select number of charts and associated analytical findings.
The QGIS sections cover the fixed route and stop overview, the population heatmap, and the population density choropleth. Each map is presented with design decisions and what it illustrates relative to the system. The density choropleth includes an animation that toggles between the city-wide view and a zoomed in neighborhood scale showing population density and total population numbers.
The NTD section walks through the data story in sequence. I lead with operating expenses and vehicle revenue hours to establish the cost and service baseline while introducing the FR/DR split. The VOMS and average fleet age chart shows that the 2020 fleet expansion coincided with COVID. The UPT chart contextualizes the difference in scale between the two modes in regards to ridership and illustrates a partial post-COVID ridership recovery. The final chart pair is OE per UPT and UPT per VRH, delivering the core reality that Demand Response costs substantially more per passenger despite lower raw operating expenditure.
The deck closes with a conclusion slide covering HSR feeder readiness, the SB-79 and AB-2011 policy context, and recommendations for potential high-frequency corridor pilots and microtransit as a DR alternative. I also included a note on per-route data being unavailable and how I'd have liked to include the data to deepen the analysis. A glossary and reference slide are included at the end for readers unfamiliar with NTD terminology.

## Conclusion and Key Findings
The data revealed a system under significant financial pressure. Operating expenses for both Fixed Route and Demand Response have risen steadily since 2014 while ridership has not recovered to pre-COVID levels (though it is on track). Ridership currently sits at half of 2014 ridership as of 2024's reporting. 
I speculate that the recovering ridership base and the potential of economic stimulation via the city's upcoming High Speed Rail Station represents an opportunity and a source of urgency. Full details of the findings and recommendations are covered in the presentation deck.
